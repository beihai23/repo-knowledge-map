# 双向迭代收敛(生成流程)

知识抽取不是单向流水线,是**带反馈回路的迭代**。深入细节会推翻/修正之前的总体认识——这对大仓库是必然,必须显式建模。

## 为什么单向不行

- **自顶向下(L0→L3)**:L0 的域划分和依赖边是"没看细节就猜的"。下探后发现域划错、依赖边错、子系统名不副实,但 L0 已"定型"。
- **自底向上(L3→L0)**:没有全局框架,海量子域无法有效聚类;并发切片连边界都没有。

两种都会让 L0 总图带着未经验证的猜测。大仓库尤其严重。

## 双向迭代收敛模型

```
骨架(L0 假设) → 下探(L1/L2/L3 取证) → 回写修正 L0 → 收敛判定
      ↑                                                    │
      └────────────── 未收敛则重下探受影响域 ────────────────┘
```

### Phase 1 — 骨架(快速 L0,低成本)
基于入口文件、目录结构、README、CLAUDE.md 产 L0 粗骨架(子系统划分 + 初步依赖边 + 技术栈)。**显式标记为"工作假设"**(用 `[假设]` 标注每条域划分/依赖边),不是定论。目的是给下探一个框架 + 给并发切片边界。

### Phase 2 — 下探(L1/L2/L3 取真实证据)
按骨架切片,逐域抽取 L1/L2/L3。重点是**收集证据**(cite):真实调用链、真实依赖、真实状态机。证据可能支持骨架,也可能反驳。

### Phase 3 — 回写修正 L0
用 Phase 2 的证据修正骨架:
- 域划分错(两域实际强耦合)→ 合并 / 重划
- 依赖边被推翻(L0 画了 A→B,但代码里没调用)→ 删边
- 子系统名不副实或不存在 → 撤销
- 漏了关键域 → 补

**回写三处同步(硬约束)**:任何节点增/删/改必须**同时**落到三处——`graph.json`(数据层)+ `INDEX.md`(展示层)+ `report.md`(修正日志)。**漏一处即未完成,不进 Phase 4。** 实测教训:曾把漏掉的子系统补进 graph.json,却没同步 INDEX(仍写"9 子系统")——数据层补了,展示层泄漏。三处同步堵这个缺口。

**cite 是修正依据**:每处修正挂反证 cite,report 记录"原假设 → 修正后 → 证据"。

### Phase 4 — 收敛判定
- 若 L0 变化超过阈值(重划的域数、删/增的边数)→ 重下探受影响域(Phase 2),进入下一轮。
- **收敛的必要条件(硬门禁)**:变化低于阈值 **且** 覆盖审计 `audit.py` 退出码 = 0(A 关孤儿清零 + C 关 cite 全过 + 文件级规模真实,见 `coverage-audit.md`)。两者皆满足才交付;audit 非 0 = 视未收敛,回 Phase 2/3 修(补孤儿 / 修 cite / 改规模数字)。
- 硬上限:最大迭代轮数(如 3 轮),避免震荡不收敛;**到上限仍未达 audit exit 0 的,取最优一轮交付,但 report 必须显式列出未清零的孤儿 / 失效 cite / 规模不符项(诚实留白,不假装收敛)**。

## 与并发模式的协调

并发下迭代更复杂,但流程一致:
1. 主 agent 产 L0 骨架(串行)→ 据此切片并发下探。
2. **整合 agent** 比对各切片证据与 L0 骨架,发现矛盾 → 回写修正 L0。
3. 若 L0 修正影响某切片 → 该切片 agent 重跑(只重跑受影响域,不全跑)。
4. 收敛判定同上。

切片 agent 不改 L0,只产切片证据;L0 的修正权在整合 agent(全局视野)。

## report 记录修正历史

report.md 必含"认识修正日志":
- 每轮:L0 哪些假设被证实、哪些被推翻、哪些重划。
- 最终:从骨架到收敛,总共修正了 N 处,引用证据 cite。
这让"知识库是怎么来的"可审计,也是质量的证明。

## 规模 gate(何时迭代)

- 小仓库(域 ≤ ~8,单 context 装得下):**单轮**(骨架→下探→一次自检)。多轮迭代是浪费。
- 大仓库(域多、跨服务、装不下):**必须迭代收敛**,否则 L0 带错。

规模判定与并发模式同步:大仓库 = 并发 + 迭代收敛;小仓库 = 串行 + 单轮。
